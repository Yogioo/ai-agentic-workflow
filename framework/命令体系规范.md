# 命令体系规范

## 1. 文档定位

本文档定义工作流中的命令分层、默认入口、内部状态机与输出约定。

目标是让普通用户只记住极少数命令，同时让 Main Agent 能根据项目状态自动决定下一步动作。

本文档同时定义一条重要分层原则：

- 面向人类的输出优先做用户优化
- 面向系统的文档优先做规范化、结构化与合规化

命令设计必须显式区分这两层，避免把系统内部状态直接当成普通用户的主输出。

当命令平台会把用户在命令后的附加文本自动拼接到命令正文末尾时，命令模板应预留清晰的“尾随输入承接位”。

尾随输入承接原则：

- 不让尾随文本孤立悬挂在命令末尾
- 为每个命令提供与其职责匹配的语义承接句
- 若尾随输入为空，则按默认逻辑执行
- 若尾随输入明确，则优先将其视为该命令的上下文补充
- 若尾随输入表达多个冲突目标，不要猜测，应先请求用户确认

---

## 2. 命令分层

### 2.1 默认用户命令

- `start`：统一启动入口，用于项目启动、项目接管、已有项目分析、文档恢复与首次规划
- `next`：统一推进入口，用于根据当前状态自动推进一步

普通用户默认只需要记住这两个命令。

### 2.2 高级命令

- `kanban`：生成或刷新当前阶段 Kanban
- `dispatch-dev`：生成 Dev Task Package
- `dispatch-qa`：生成 QA Task Package
- `status-sync`：根据回传结果更新状态建议

高级命令主要面向熟悉工作流的人类或 Main Agent 自身使用。

高级命令的输出也应尽量稳定、结构化，便于：

- 人类快速判断结果是否可用
- Main Agent 继续推进后续动作
- 必要时同步到 `当前状态快照.md` 或正式项目文档

---

## 3. start 命令定位

`start` 是统一项目入口，用来处理以下场景：

- 新项目，只有模糊想法
- 已有部分代码，但没有文档
- 有文档但不完整
- 中断后重新接管项目
- 不确定当前应该先做什么

`start` 的职责不是直接开始编码，而是先判断当前项目状态，再把项目带到“可以稳定推进”的状态。

`start` 是重入口命令，适合在以下场景使用：

- 项目刚初始化后第一次进入
- 新会话第一次接管项目
- 用户不清楚当前项目状态
- 文档与代码明显失配，需要重新判断状态

`start` 不应被当作每轮开发前的日常命令。

---

## 4. next 命令定位

`next` 是统一推进命令，用来在初始化之后推动项目前进。

它每次只推进一步，不同时执行多个大动作。

`next` 的目标是：

- 根据当前文档与任务状态，选择最合理的下一步
- 输出当前所处阶段、当前缺口、推荐动作与预期产物
- 在必要时调用高级命令对应的流程，但不要求用户显式记忆这些命令

`next` 是轻入口命令，适合在 `start` 完成后作为日常默认入口反复使用。

`next` 应优先做最小增量读取，而不是重新进行全量项目诊断。

---

## 4.1 读取原则

命令执行时，应区分三类信息：

- 角色规则：由当前 Agent 的系统提示词承担，不应在每次命令中重复读取
- 命令规则：由 `命令体系规范.md` 提供
- 项目状态：由当前项目文档和必要的局部代码上下文提供

因此：

- `start` / `next` 不应把 `MainAgent.md` 当作日常重复读取对象
- `start` / `next` 不应默认把 `总览入口.md` 当作每次必读文档
- `start` / `next` 应优先读取项目状态文档，而不是角色定义文档

---

## 4.2 start 读取策略

`start` 允许较重读取，但应尽量聚焦于“状态判断”所需内容：

1. `framework/命令体系规范.md`
2. `当前状态快照.md`（若存在）
3. `当前PRD.md`（若存在）
4. `当前RoadMap.md`（若存在）
5. `当前阶段Kanban.md`（若存在）
6. `BUG追踪.md`（若存在）
7. 必要的局部代码或项目结构信息

只有在状态判断确实需要时，才进一步扩展读取范围。

`start` 在完成状态识别或补全文档后，应更新 `当前状态快照.md`。

## 4.3 next 读取策略

`next` 应优先执行最小增量读取：

1. `当前状态快照.md`
2. `当前阶段Kanban.md`
3. `当前RoadMap.md`
4. 必要时再读取 `当前PRD.md`
5. 仅在涉及缺陷、回归或阻塞时读取 `BUG追踪.md`
6. 仅在当前动作需要代码事实时，读取最小必要代码上下文

`next` 不应重复做全量项目接管分析。

若 `当前状态快照.md` 不存在、明显过期，或与正式文档冲突，则应以正式文档为准，并在本轮结束时刷新 `当前状态快照.md`。

---

## 5. start 内部状态识别

`start` 应优先识别当前项目处于以下哪种状态：

- `S0_EMPTY`：几乎没有代码，也没有当前文档
- `S1_IDEA_ONLY`：用户有目标描述，但项目文档未建立
- `S2_CODE_NO_DOC`：已有代码，但缺 PRD、RoadMap 或 Kanban
- `S3_PRD_MISSING`：有代码或部分文档，但 `当前PRD.md` 不完整
- `S4_ROADMAP_MISSING`：PRD 已有，但 RoadMap 不完整
- `S5_KANBAN_MISSING`：PRD、RoadMap 已有，但当前阶段任务未拆
- `S6_READY_TO_EXECUTE`：文档已基本齐备，可以进入执行推进

这些内部状态主要用于系统判断，不应直接作为面向普通用户的主输出。

---

## 5.1 面向用户的阶段话术

对普通用户输出时，应优先使用自然语言阶段描述，而不是状态码。

推荐映射如下：

- `S0_EMPTY` / `S1_IDEA_ONLY`：项目还没真正开始，我们先把想法和目标讲清楚
- `S2_CODE_NO_DOC`：项目已经有一些内容了，但还没把需求和规划整理清楚
- `S3_PRD_MISSING`：我们现在处于“明确需求”的阶段
- `S4_ROADMAP_MISSING`：我们现在处于“整理开发规划”的阶段
- `S5_KANBAN_MISSING`：我们现在处于“拆分当前阶段任务”的阶段
- `S6_READY_TO_EXECUTE`：准备工作已经差不多了，可以开始推进具体任务了

除非用户明确要求查看详细状态，否则不要把 `S0`、`S1`、`PRD_BUILD` 之类内部术语直接作为主输出。

---

## 6. start 推荐动作映射

- `S0_EMPTY` / `S1_IDEA_ONLY`：先做需求发掘，生成或补全 `当前PRD.md`
- `S2_CODE_NO_DOC`：先分析现有项目，再反推 `当前PRD.md` 与 `当前RoadMap.md`
- `S3_PRD_MISSING`：优先补全 `当前PRD.md`
- `S4_ROADMAP_MISSING`：优先补全 `当前RoadMap.md`
- `S5_KANBAN_MISSING`：优先生成 `当前阶段Kanban.md`
- `S6_READY_TO_EXECUTE`：输出当前状态摘要，并建议转入 `next`

---

## 7. next 内部推进状态

`next` 应优先把当前项目映射到以下推进状态：

- `DISCOVER`
- `PRD_BUILD`
- `ROADMAP_BUILD`
- `KANBAN_BUILD`
- `TASK_DISPATCH_READY`
- `DEV_IN_PROGRESS`
- `QA_IN_PROGRESS`
- `STATUS_SYNC`
- `PHASE_CLOSE`
- `NEXT_PHASE`

这些推进状态主要用于内部状态机，不应直接作为普通用户的主输出。

---

## 7.1 面向用户的推进话术

推荐将推进状态翻译为以下人类可读表达：

- `DISCOVER`：先把项目情况和目标聊清楚
- `PRD_BUILD`：先把需求整理清楚
- `ROADMAP_BUILD`：先把开发阶段规划出来
- `KANBAN_BUILD`：先把当前阶段任务拆出来
- `TASK_DISPATCH_READY`：可以开始安排第一个具体任务了
- `DEV_IN_PROGRESS`：当前正在推进开发实现
- `QA_IN_PROGRESS`：当前正在做验证和检查
- `STATUS_SYNC`：当前正在同步结果和项目状态
- `PHASE_CLOSE`：当前阶段接近完成，准备收口
- `NEXT_PHASE`：准备进入下一阶段

---

## 8. next 推荐迁移

- `DISCOVER -> PRD_BUILD`
- `PRD_BUILD -> ROADMAP_BUILD`
- `ROADMAP_BUILD -> KANBAN_BUILD`
- `KANBAN_BUILD -> TASK_DISPATCH_READY`
- `TASK_DISPATCH_READY -> DEV_IN_PROGRESS`
- `DEV_IN_PROGRESS -> QA_IN_PROGRESS` 或 `STATUS_SYNC`
- `QA_IN_PROGRESS -> STATUS_SYNC`
- `STATUS_SYNC -> TASK_DISPATCH_READY` 或 `PHASE_CLOSE`
- `PHASE_CLOSE -> NEXT_PHASE`
- `NEXT_PHASE -> KANBAN_BUILD`

`next` 每次只应推进一步，并明确说明为何选择这一步。

但 `next` 不应无条件自动穿越所有状态。

---

## 8.1 确认闸门

`next` 应区分两类节点：

- 自动推进节点：可以直接继续执行
- 确认闸门节点：必须先请求用户确认，再继续推进

推荐确认闸门如下：

- `GATE_PRD_CONFIRM`：PRD 草案已形成，需用户确认需求是否成立
- `GATE_ROADMAP_CONFIRM`：RoadMap 草案已形成，需用户确认阶段规划是否成立
- `GATE_EXECUTION_START`：准备从规划进入开发执行，需用户确认是否开始第一轮开发
- `GATE_PHASE_CLOSE`：当前阶段接近完成，需用户确认是否收口
- `GATE_NEXT_PHASE`：准备进入下一阶段，需用户确认是否切换阶段

`next` 在遇到确认闸门时，不应继续自动推进，而应停下来输出：

- 当前已经完成了什么
- 下一步如果继续会发生什么
- 为什么这个节点需要用户确认
- 用户现在可以怎么选择

---

## 8.2 自动推进与确认边界

以下动作通常可以由 `next` 自动推进：

- 补全或刷新草稿性文档
- 生成 Kanban 草案
- 生成任务包草案
- 汇总结果并给出状态同步建议
- 刷新 `当前状态快照.md`

以下动作通常不应默认自动穿越：

- 将需求草案视为已确认需求
- 将 RoadMap 草案视为已确认规划
- 未经确认就从规划阶段直接进入开发派单
- 未经确认就结束当前阶段
- 未经确认就切换到下一阶段

---

## 8.3 auto 命令模式

`auto` 是在 `next` 之上的自动推进策略命令。

它本质上是：

- 循环执行 `next`
- 根据自动化模式决定是否跨过确认闸门
- 在达到停止条件时停下并汇报

当前支持 3 种模式：

- `task`：自动推进到当前任务完成后停止
- `phase`：自动推进到当前阶段完成后停止
- `sisyphus`：持续自动推进，直到项目完成、遇到阻塞或用户中断

当命令平台支持把用户在命令后的附加文本直接拼接到命令提示中时，`auto` 必须显式从该附加文本中解析模式参数。

解析规则建议固定为：

- 若附加文本中明确包含 `task`，则模式为 `task`
- 若附加文本中明确包含 `phase`，则模式为 `phase`
- 若附加文本中明确包含 `sisyphus`，则模式为 `sisyphus`
- 若未提供明确模式，则默认使用 `phase`
- 若附加文本同时包含多个模式关键词或语义冲突，则不要猜测，应先请求用户确认

### `task` 模式

- 适合最保守自动化
- 在以下情况停止：
  - 一个 Dev 任务完成
  - 一个 QA 审查完成
  - 一个状态同步动作完成
  - 遇到阻塞

### `phase` 模式

- 推荐默认模式
- 适合让 AI 自动推进当前阶段，但在阶段边界停下来等人类 review
- 在以下情况停止：
  - 当前阶段完成
  - 命中阶段收口相关确认闸门
  - 遇到阻塞

### `sisyphus` 模式

- 适合最大化自动推进
- 在以下情况停止：
  - 项目完成
  - 遇到阻塞
  - 用户中断
  - 达到安全护栏（如连续无进展）

---

## 8.4 auto 与确认闸门的关系

- `task` 模式：默认不跨过关键确认闸门，停下来等待用户确认
- `phase` 模式：允许在阶段内部自动推进，但在阶段收口和跨阶段节点停下
- `sisyphus` 模式：可跨过大多数确认闸门持续推进，但仍应在阻塞、无进展或高风险情况下停下

`auto` 不是替代 `start` 的冷启动命令，而是建立在 `start` / `next` 之后的自动推进命令。

---

## 9. 最小提问原则

- 每次最多只问 1 到 3 个真正影响状态迁移的问题
- 能从代码、文档或任务包推断的内容，不再追问用户
- 不影响当前一步推进的问题，应延后
- 对已有代码项目，应优先从代码中提取事实，再向用户确认目标差异

---

## 10. 输出约定

先遵守以下总原则：

- 面向普通用户的主输出，应优先用户友好、自然、易懂
- 面向系统同步的附加输出，应优先结构稳定、字段明确、便于后续状态更新
- 不要把内部状态码、推理过程和同步细节直接作为普通用户的主输出主体

### 10.1 start 输出

必须至少包含：

- 当前项目状态判断
- 已读取的关键信息
- 当前最关键缺口
- 建议立即进入的下一步
- 是否更新了 `当前状态快照.md`

推荐固定输出结构：

- 状态判断
- 为什么这么判断
- 下一步建议
- 你可以怎么继续

主输出应优先面向普通用户，使用浅显自然语言，不要堆叠系统术语。

为了让用户一眼看懂重点，`start` 的默认主输出还应满足：

- 总长度尽量控制在 6 到 10 行以内
- 先给结论，再给原因，再给动作
- 不在主输出中列出读取过的全部文档
- 不在主输出中展示绝对路径
- 不在主输出中逐项回显快照字段
- 不在主输出中详细解释“修正快照”的过程
- 如果需要用户输入，优先直接给 1 到 3 个简单问题，而不是先铺陈系统分析

详细状态与快照字段可以保留，但应降级为附加信息，而不是主输出主体。

其中附加的 `快照更新` 应尽量明确给出以下字段：

- 当前项目状态
- 当前推进状态
- 当前阶段
- 当前主目标
- 当前最关键缺口
- 推荐下一步动作

### 10.2 next 输出

必须至少包含：

- 当前所处阶段
- 本次推进动作
- 推进后预期产物
- 是否更新了 `当前状态快照.md`

推荐固定输出结构：

- 当前阶段
- 为什么现在做这一步
- 接下来会得到什么
- 你可以怎么继续

主输出应优先面向普通用户，使用浅显自然语言，不要堆叠系统术语。

为了让用户快速继续，`next` 的默认主输出还应满足：

- 总长度尽量控制在 5 到 8 行以内
- 先说明现在在哪一步，再说明现在先做什么
- 不在主输出中罗列内部状态机字段
- 不在主输出中逐项回显快照字段
- 重点放在“现在做什么”和“做完会得到什么”

如果当前命中确认闸门，则主输出应改为“确认提示”模式，重点说明：

- 当前已经完成了什么
- 如果继续会发生什么
- 为什么现在需要你确认
- 你可以如何选择

详细状态与快照字段可以保留，但应降级为附加信息，而不是主输出主体。

其中附加的 `快照更新` 应尽量明确给出以下字段：

- 当前项目状态
- 当前推进状态
- 当前阶段
- 当前主目标
- 当前最关键缺口
- 推荐下一步动作

### 10.3 kanban 输出

推荐固定输出结构：

- 当前阶段
- Kanban 结果
- 并发与冲突
- 后续建议

其中应尽量明确给出：

- 阶段目标
- 任务列表
- 每个任务的执行角色
- 路径范围
- 并发 / 串行关系
- 是否需要 QA

### 10.4 dispatch-dev 输出

推荐固定输出结构：

- 目标任务
- Task Package
- 契约检查
- 后续建议

其中应尽量明确给出：

- Task ID
- 任务目标
- 接口契约
- 验收标准
- 测试用例
- 文件路径范围
- 风险提示

### 10.5 dispatch-qa 输出

推荐固定输出结构：

- 目标任务
- QA Task Package
- 风险判断
- 后续建议

其中应尽量明确给出：

- Task ID
- 审查目标
- 验收标准
- 测试与审查要求
- 文件路径范围
- 风险提示

### 10.6 status-sync 输出

推荐固定输出结构：

- 当前回传结果
- 状态更新建议
- 是否可继续推进
- 快照 / 文档同步建议

其中应尽量明确给出：

- 受影响的 Task ID
- 当前建议状态
- 更新依据
- 是否已通过验收
- 是否需要 QA 或重新派单
- 下一步建议

---

## 11. 实施原则

- `start` 与 `next` 是默认入口，优先级高于高级命令
- 高级命令应作为 `start` / `next` 的内部能力或人工高级入口存在
- README 与对外说明文档应优先教用户使用 `start` 与 `next`
